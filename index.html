An Open Manifesto

Git was for humans.
We need something else.

Everyone nowadays runs engineering teams where agents write more code than engineers do. The old workflow (checkout, branch, push, PR, wait for review, rebase, pray) was designed for human speed. It's breaking. Not slowly. Right now. This is a first attempt at figuring out what replaces it. I am not rebuilding Git. I am building a new primitive for how code gets written in a world where the writers are machines.

01 / The Real Problem

Git has always been broken for code

Agents didn't create this problem. They just made it impossible to ignore.

Every developer has lived this: you open a PR, CI is green, no merge conflicts, you merge confidently, and you break production. Because someone else's PR, also green, also conflict-free, changed a function signature or shifted what a return value means in something you depend on. Git told you everything was fine because Git checks lines, not meaning.

Look at the whole ecosystem of band-aids we've built. CI pipelines, required status checks, "rebase on main before merge" policies, integration branches, merge queues. All of these exist because Git's merge model is semantically blind and everyone knows it. We've just normalized the pain. When a senior dev "knows" which files to check after a merge, that's a human doing semantic dependency resolution in their head because the tool won't do it.

The difference with agents is purely scale. A human team merges maybe 5-10 PRs a day, and the odds of a semantic collision are low enough that CI catches most of it. When 20 agents are committing every few minutes, the combinatorial explosion of potential semantic conflicts makes the band-aid approach collapse. You can't run full CI on every commit at that velocity, and you can't have a human reviewer eyeballing each merge for subtle dependency breaks.

So the framing shifts. This isn't "Git is broken for agents." It's something more fundamental: version control that treats code as text was always the wrong abstraction. We tolerated it because at human velocity, the failure rate was manageable. Agents just removed that safety margin.

02 / The Collapse

The filesystem is a terrible primitive for agentic code

Think about what Git actually assumes. One person. One machine. One local copy of the codebase. You work on it, you push it, someone reviews it. That was elegant when the bottleneck was how fast a human could think and type.

Now picture 50 agents modifying the same codebase at the same time. You don't get "merge conflicts." You get chaos. Branch-per-agent gives you thousands of divergent realities. Filesystem locks are too blunt. And PR review? That's the single-threaded bottleneck choking a massively parallel system. Your fastest engineer is now your slowest reviewer.

Here's the thing most people haven't internalized yet: code is no longer something humans carefully craft and contemplate. Code is high-velocity data. It should be stored, synchronized, and validated like data, with write atomicity, real-time subscriptions, conflict resolution, and continuous validation baked into the storage layer. Not bolted on through CI pipelines that run 20 minutes after the fact.

03 / The Thesis

Store code in a database, not a filesystem

This is the core idea. Treat the codebase as a live, transactional, event-sourced data system. Not a tree of files that get synchronized through patches and diffs.

Every write is a transaction. An agent acquires a fine-grained lock at the function level or the block level, not the whole file, makes its change, and it's immediately visible to every other agent. No more "push and pray." No more rebasing a 78-commit branch against a target that moved while you were working.

Every mutation is replayable. This is the part that matters most. Humans couldn't externalize their thought process while coding. They just... wrote code and left a commit message. Agents can capture everything: the reasoning, the alternatives they considered, the context they consumed, the confidence level. Your version history stops being a flat diff log and becomes a complete decision graph. That was never possible before.

Review happens continuously, not at a gate. Real-time validation (lint, format, type-check, test, security scan) runs on every single transaction. Not in some CI pipeline that fires 20 minutes later. Review shifts from "human sitting in the critical path" to "continuous automated attestation with human oversight on the things that actually matter."

04 / What Dies

Concepts that don't survive the transition

The Branch. Branching exists because workers are disconnected from each other. When all agents operate against a shared transactional store, you don't need branches for daily work. You might still branch for experimentation, like trying a radically different approach in isolation, but it stops being the default mode for getting anything done.

The Pull Request. A PR is basically a batch review of accumulated work. When every mutation gets validated in real-time and the full reasoning chain is captured alongside it, the whole "request permission to merge" ceremony becomes unnecessary. Humans set the policies. The system enforces them. Done.

The Merge Conflict. Conflicts exist because of disconnected state. Period. A transactional system with fine-grained locking and CRDTs resolves conflicts at write-time. Two agents editing the same function negotiate in real-time instead of discovering the problem three days later through a three-way diff.

The Local Checkout. There is no "local." The codebase is a shared distributed store. Agents read and write directly. Humans interact through views (IDE integrations, dashboards, policy consoles) that project the live state. The mental model shifts from "I have my copy" to "I have my view."

05 / Principles

The 9 principles of agent-native version control

01

Code is data, not files

The codebase is a structured, queryable, transactional data store. Files are a projection, not the source of truth. Think Postgres, not ext4.

02

Write-level atomicity

Every code mutation, from a single-line change to a multi-file refactor, is an atomic transaction with ACID guarantees. No partial states. No broken intermediaries.

03

Fine-grained coordination

Lock at the function, block, or symbol level. Not the file. Agents negotiate concurrent access through the storage layer, not through merge resolution after the fact.

04

Real-time subscriptions

Agents and tools subscribe to codebase state changes. When agent A modifies a function signature, agent B (working on a caller) is notified instantly and can adapt.

05

Continuous validation

Lint, format, type-check, test, and security scan run on every transaction. Not in a CI pipeline minutes later. Invalid code is rejected at write-time.

06

Captured reasoning

Every mutation carries structured metadata: the agent's intent, alternatives considered, context consumed, confidence level. The version history is a decision graph, not a diff log.

07

Policy, not process

Humans define policies (security boundaries, architectural constraints, review thresholds). The system enforces them continuously. The human role shifts from reviewer to governor.

08

Full replayability

Any codebase state, including the reasoning and context that produced it, can be reconstructed at any point. Time-travel debugging across the entire project history, including agent thought processes.

09

Human-legible views

The underlying store is agent-optimized. Humans interact through projections: IDE integrations, dashboards, policy consoles, natural language queries against the codebase.

06 / Architecture

A strawman stack

Storage Layer
A transactional database (Postgres, CockroachDB, or a purpose-built store) holding the codebase as structured data. Tables for files, symbols, dependencies, mutations. Every write is a transaction with row-level or finer locking. Event log (WAL-inspired) for full replayability.
Coordination Layer
Manages agent sessions, lock acquisition, conflict detection, and real-time subscriptions. Implements operational transforms or CRDTs for concurrent edits to the same region. Pub/sub for codebase state change notifications. Think: a multiplayer engine for code.
Validation Layer
Runs continuously on every committed transaction. Pluggable pipeline: parse → lint → format → typecheck → test → security. Invalid transactions are rejected or flagged. Replaces CI/CD for the inner loop entirely.
Reasoning Layer
Captures and indexes agent intent, alternatives, context, and confidence alongside every mutation. Enables queries like: "Why was this function rewritten?" or "Show me every change made with confidence below 0.7." This is what humans couldn't produce but agents can, and it's transformative for auditing and debugging.
Policy Layer
Human-defined rules enforced at write-time. Architectural boundaries ("no direct DB calls from the API layer"), security policies, review thresholds ("changes to auth require human approval"), resource budgets. The governance plane for agentic codebases.
Projection Layer
Translates the live data store into human-friendly views. IDE plugins that render the current state as a familiar file tree. Dashboards showing agent activity, validation status, policy compliance. Natural language interfaces for querying codebase history and intent.

08 / The Design

Seven foundational decisions

01 / Unit of Work

Two primitives, not one

Git's "commit" is a snapshot of the entire repo. That made sense when a human would spend 30 minutes making a coherent set of changes. For agents, this is absurd. An agent might change one line in one function, or refactor 40 files in 3 seconds. The commit is either too coarse or too fine.

The atomic unit is the semantic mutation. A single meaningful change to a single semantic element. Not "line 47 changed from X to Y" but "function calculateTotal signature changed from (items: Item[]) -> int to (items: Item[], currency: Currency) -> Decimal." Each mutation is a node in a graph with a type (create, modify, delete, rename, move) and a target (function, type, interface, constant, import).

The logical unit is the task. A directed acyclic graph of semantic mutations that together accomplish something. "Implement multi-currency support" is a task of 15 mutations across 8 files. Tasks are what agents get assigned, what humans review, and what gets rolled back if something breaks. Like a PR, but a first-class entity in the data model, not a diff between two branch tips.

Why two levels matters: the system can accept or reject individual mutations based on semantic conflicts, while humans reason about tasks. An agent's task might have 14 clean mutations and 1 conflicting one. Instead of rejecting the whole thing, the system pinpoints exactly which mutation conflicts and why. The agent fixes just that one.

The thing Git gets catastrophically wrong: a commit bundles unrelated changes into one blob. "Fixed the bug AND ran the linter AND renamed that variable" as one unit. Need to revert just the bug fix? Good luck. In this system, those are three separate mutations inside one task, independently addressable.

02 / Coordination

Serializable Snapshot Isolation over a semantic dependency graph

Step 1, Task assignment. An agent gets a task. The system computes the likely impact zone, the set of semantic elements the agent will probably touch, based on the task description and the dependency graph. This is an estimate, not a lock.

Step 2, Snapshot. The agent gets a consistent snapshot of the impact zone plus its transitive dependencies. Not the whole repo, just the subgraph it needs. Cheap because you're querying a database, not cloning a repo.

Step 3, Work. The agent works against its snapshot. Generates semantic mutations. No coordination overhead. Fully independent.

Step 4, Commit validation. Three layers:

  • Structural conflict. Did another agent modify the same semantic element since our snapshot? Hard conflict. First-writer-wins or priority-based.
  • Interface contract. Did any element I depend on change its interface? If I'm calling formatPrice(amount) but another agent changed it to formatPrice(amount, locale), the dependency graph catches this instantly. No compile needed.
  • Behavioral validation. Does the combined state pass relevant tests? Expensive, so it runs on task completion. Because you have the dependency graph, you run targeted tests, not the full suite.

Why this beats the alternatives: Locks serialize agents. CRDTs can't work because code merges aren't commutative (addParameter(x) and removeFunction(x) don't compose). OT at the character level spends more compute on transforms than on writing code, and still misses semantic conflicts.

The throughput math: 1000 semantic elements, 20 agents, each task touching about 10 elements. Probability of structural conflict per commit is roughly 19%. Interface conflicts are slightly higher due to transitive dependencies, but detection is O(edges) not O(codebase). Near-linear scaling until the dependency graph becomes so interconnected that most impact zones overlap. At that point you have an architecture problem, not a tooling problem.

03 / Consistency

Strong where it matters, eventual where it doesn't

The knee-jerk answer is "strong consistency, obviously, code has to be correct." That's wrong at the scale we're targeting.

Strong consistency within a dependency subgraph, eventual consistency across independent subgraphs. If Agent A is working on payments and Agent B is working on the onboarding flow, and those two subgraphs share no semantic dependencies, they don't need to see each other's changes in real-time. They're working in effectively different codebases that happen to live in the same repo.

But if Agent A changes a shared utility function that Agent B depends on, Agent B needs to know before it commits, not after. The dependency graph naturally partitions into clusters of tightly-coupled code connected by thinner edges. Within a cluster, strong consistency. Across clusters, eventual consistency with conflict detection at commit time.

The tricky edge case: shared utilities. Every codebase has a utils.ts that everything depends on. If you make that a single serialization point, it becomes a bottleneck. The fix: decompose at function-level granularity. utils.formatDate and utils.formatCurrency are independent semantic elements. Two agents can modify different utility functions concurrently because the graph tracks dependencies at the function level, not the file level.

Streaming validation: instead of discovering conflicts at commit time (wasting 5 minutes of work), the system continuously checks whether an agent's snapshot dependencies are still valid. If something changes, the agent gets an interrupt mid-task and can adapt. This is a massive advantage over Git where you only discover conflicts when you try to merge.

04 / Reasoning Traces

Code that knows why it exists

This is the moat. Every other part of this system is hard engineering. The reasoning trace is what makes this genuinely new.

The problem with "just store the LLM's reasoning": an agent producing 100 mutations per minute, each with a paragraph of reasoning, generates gigabytes of unstructured text per day. Nobody will read it. Unsearchable. Mostly noise.

Tier 1, Structured metadata (always stored, machine-readable). Compact and queryable: intent (feature, bugfix, refactor, optimize, adapt-to-dependency-change), confidence (float), alternatives_considered (count), dependencies_read, constraints_applied. About 200 bytes per mutation. At 1000 mutations/min, that's roughly 288MB/day. Trivial.

Tier 2, Structured rationale (stored for non-trivial mutations). 1-2 sentences for why this approach, why not alternatives, and explicit assumptions. About 500 bytes per mutation, stored for roughly 30% of mutations. About 100MB/day.

Tier 3, Full reasoning dump (ephemeral, garbage-collected). The raw LLM chain. Useful for forensics when things go badly wrong. About 5KB per mutation, 7-day retention. Roughly 49GB rolling. Manageable in cold storage.

The killer query: "Why does this function exist?" No human codebase can answer that today. You grep commit messages, PR descriptions, Slack threads, Jira tickets. In this system, you traverse the reasoning trace: created by Task X, because of reasoning Y, depends on A and B, assumptions are C. Transformative for onboarding, debugging, and refactoring.

05 / Human Interaction

Humans govern. Agents execute.

This is where most "AI-first" tools fail. They build for the machine and bolt on a human interface as an afterthought. But humans are the principal. Agents are the executors.

At 1000+ mutations per minute, line-by-line code review is dead. Humans operate at three levels instead:

  • Policy definition. Before any agent writes code, humans define machine-enforceable constraints. "All API endpoints validate input with Zod schemas." "No new dependencies over 100KB." "No changes to auth without human approval." The system checks these on every mutation.
  • Task-level review. Humans review completed tasks, not individual mutations. Presented as: what changed (semantic summary, not a diff), why (aggregated reasoning traces), test results, confidence scores, assumptions. This is PR review at the right abstraction level.
  • Anomaly-driven intervention. The system surfaces risk: low confidence mutations, tasks with too many retries, architectural violations that pass tests, cascading dependency changes. Humans jump in when flagged, not on a fixed cadence.

The UI is not a diff viewer. It's an air traffic control dashboard: active tasks, agent utilization, conflict rates, test health, policy violations, attention queue. Humans direct traffic, not read every line.

Escalation granularity matters: hard blocks (agent cannot proceed), soft warnings (proceed but flagged), async follow-ups (create a task to fix it later). Getting this right is what makes the system usable instead of either too permissive or too rigid.

06 / Migration

A bridge to Git, not a wall

Every potential user has an existing Git repo. If the system requires a clean-room start, adoption is dead.

The bridge: ingest from and export to Git as an I/O adapter. Import a Git repo, parse into semantic graph (tree-sitter + LSP for the AST, build dependency graph from there). Export back, generate conventional Git commits with readable diffs and commit messages synthesized from reasoning traces. Teams adopt incrementally: agents work in the new system, but compliance/legal still sees a Git repo updated via the bridge.

The long-term play: once teams see semantic conflict detection, reasoning traces, and task-level review, they stop looking at the Git export. It becomes an artifact for backward compatibility, like PDF invoices in a digital-first company. Eventually, the new system is the source of record.

07 / Recovery

Semantic rollback, not diff reversal

Agent-level failure: an agent crashes mid-task. Every mutation is individually recorded, so there's no "half-committed" state. The completed mutations are valid. The task is marked incomplete. Another agent picks it up, reads the existing mutations and reasoning traces, and continues. Try doing that with a half-finished Git branch from a developer who left.

Semantic regression: changes pass all validation but cause a production issue. The system traces the causal chain: this task, these mutations, these semantic elements, these code paths. Rollback is targeted. Revert the specific mutations, not "revert the whole commit and the 5 commits after it because Git made them sequential."

Cascading failure: Agent A makes a change, agents B, C, D adapt to it, then A's change gets rolled back. B, C, D's adaptations are now orphaned. The dependency graph detects this immediately: A's rollback invalidates downstream mutations with adapt_to_dependency_change intent linked to A's mutation. Flagged for re-evaluation automatically.

The key insight: in Git, rollback is "reverse the diff." Here, rollback is "undo this decision and everything downstream of it." That's the difference between text-aware and semantics-aware version control.
