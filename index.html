g4a

$ the reasoning layer for AI-written code

Git stores what changed. The reasoning behind it is lost. g4a captures it.

Add it to your existing project. Change nothing. Unlock 10x for both humans and AI agents.

Install
Step 1 - install g4a (one time, global)
brew install lcbasu/g4a/g4a
Step 2 - initialize in each repo
cd your-project && g4a init

That's it. Use your AI agent normally, make commits. Reasoning is captured automatically.

How to use

g4a log
See the full reasoning chain for every commit. Every file the agent read, every approach it tried, every dead end, every decision. The complete story behind the code.
g4a why "auth"
Search the decision trail by file, function, or keyword. Find why any decision was made, across any commit, by any agent. Try g4a why "database migration" or g4a why payment.py.
g4a log --short
Compact summary view when you just need the highlights - what changed, what the agent intended, which files were touched.

Works for everyone, forever

Everything is stored in git. No external server. No database. No account. No hosting.

Any developer on any system pulls the latest commits and gets the full reasoning history. Any future AI agent reads it before modifying shared code. It works through GitHub, GitLab, Bitbucket, or any internal git host at Google, Amazon, or anywhere else that runs git.

The reasoning travels with the code. Fork the repo, the reasoning forks too. Clone it in 5 years, the reasoning is still there. Zero external dependencies. Everything lives in git.

The problem

Every AI coding agent thinks before it writes. Claude Code reads 20 files, considers 3 approaches, rejects 2, tests edge cases, and picks the best path. Cursor explores your codebase, reasons about patterns, and adapts. Codex plans multi-file changes before executing them.

Then at commit time, all of that thinking is thrown away.

This happens with every agent, every commit, every tool. The reasoning is generated, used once, and discarded. The most valuable artifact of the development process - the WHY behind every decision - is lost forever.

What this looks like in practice

Your agent refactors the payment processing module. The commit says:

What the reviewer sees in the PR

refactor: Update payment calculation to use Decimal

8 files changed. Float replaced with Decimal in checkout, billing, refunds, and settlement. Some rounding logic changed. Tests updated.

Why? No idea. Was it a precision bug? A compliance requirement? Did the agent know about the nightly settlement job? Did it test batch operations?

The reviewer spends 25 minutes reading 8 diffs, checking rounding logic, and ultimately approves because the tests pass and they're behind on 14 other PRs.

three weeks later

The nightly settlement job is off by $0.03 on 10,000 transactions. $300 missing. Finance escalates.

Someone runs git blame. Finds the refactor commit. Reads "Update payment calculation to use Decimal." Now what? The agent session is gone. Its reasoning is gone. All that's left is 8 diffs and a commit message.

same commit, with g4a
What g4a captured

Intent: Switch from float to Decimal for currency precision. After testing with 10,000 simulated transactions, float arithmetic drifted by $0.03 while Decimal was exact.

Exploration: Read checkout.py, billing.py, refunds.py, settlement.py. Found 14 call sites. Ran test with 10,000 simulated transactions. Checked batch_settlement_job.py - it calls calculate_total() which now returns Decimal.

Alternatives: (1) Keep float + round at the end - rejected, error accumulates across batch ops. (2) Integer cents - rejected, existing APIs expect decimal format, would touch 23 files. (3) Decimal everywhere - chosen, cleanest path, 8 files.

Risk: batch_settlement_job.py CSV export on line 47 uses f-string formatting that may truncate Decimal. Flagged LOW confidence.

Confidence: 0.85 overall, 0.6 on CSV export formatting

Files read: checkout.py, billing.py, refunds.py, settlement.py, batch_settlement_job.py, tests/

Without g4a

25-minute review, missed risk, $300 settlement error three weeks later, hours of debugging with no trail.

With g4a

3-minute review, spotted the LOW confidence flag on CSV export, fixed the one line, approved. Settlement works perfectly. The reasoning is in the repo forever.

How g4a helps humans

PR reviews go from guessing to reading
Instead of reverse-engineering intent from diffs, reviewers read the agent's actual reasoning. Intent, confidence, alternatives, what was tested, what was flagged as risky. A 25-minute review becomes 3 minutes with higher confidence.
Debugging has a trail
When something breaks, g4a why settlement_job gives you the complete decision history. Every change, every agent, what they intended, what they were worried about. No more "who changed this and why?"
Trust is earned, not assumed
Every AI-generated change carries a confidence score and full explanation. Low-confidence changes get scrutiny. High-confidence changes with thorough exploration can be approved faster. Trust becomes data-driven.
Onboarding is instant
New developer joins the team. Instead of weeks figuring out why the codebase looks this way, they run g4a why <module> and read the decision history. The codebase documents itself.

How g4a helps AI agents

Every future agent will have context
The .g4a/ directory is institutional memory. A new agent reads it and immediately knows: this was changed to Decimal because of precision drift in batch settlements. Integer cents was rejected because it would touch 23 files. No re-exploration. No accidentally reverting a deliberate choice.
Agents stop duplicating work
Without g4a, every agent re-discovers the same things. Agent A found that the payment module has a tricky dependency on the settlement job. Agent B comes along the next day and has to discover that from scratch. With g4a, Agent B starts where Agent A left off.
Agent coordination becomes possible
When multiple agents work on the same codebase, they need shared context to avoid conflicts. g4a provides the foundation: captured reasoning that any agent can read before modifying shared code. As agent-to-agent coordination matures, the reasoning is already there.
Your codebase compounds intelligence
Every decision, every rejected alternative, every risk assessment - accumulated over months. The longer g4a runs, the more context every future agent has. Your codebase goes from a collection of files to a documented history of decisions.

The solution

g4a captures the reasoning that AI agents already produce and stores it alongside the code in git. Add it to your existing project. Change nothing about your workflow.

# install (one command) brew install lcbasu/g4a/g4a # then in any repo cd your-project g4a init # use your AI coding agent normally - nothing changes # reasoning is captured automatically in the background # see the full reasoning chain for every commit g4a log # search the decision trail g4a why process_payment

Designed for all AI coding agents

g4a is not tied to any single agent. It's a universal reasoning layer that works with every tool that writes code.

Claude Code launching first
Deepest integration. Claude Code already saves full session transcripts with every file read, tool call, and reasoning statement. g4a parses these directly - no API call needed, no workflow change. The reasoning is already there. g4a just captures it before it's lost.
Cursor, Codex, Copilot, Windsurf, Aider git hook fallback
g4a installs a standard git post-commit hook. When any agent commits code, g4a reads the diff and context, then infers reasoning. Labeled "inferred" vs "captured" - weaker than direct integration, but dramatically better than nothing.
Custom agents and frameworks agent-agnostic schema
The reasoning record schema is the same regardless of which agent produced it. LangChain, CrewAI, custom pipelines - any tool that writes code through git can have its reasoning captured.

How it stores data

Reasoning lives in .g4a/ inside your repo. No external server. No account. No hosting.

Format One file per commit, one file per session. CBOR (IETF RFC 8949) + zstd compression. An internet standard, not a proprietary format.
Security Credentials, API keys, passwords, and tokens are detected and masked before any data hits disk. Irreversible.
Size 10-50 KB per commit. 1,000 commits = 10-50 MB. Within every hosting platform's limits.
Portability Clone the repo, get the reasoning. Fork it, reasoning forks too. Delete .g4a/, still a valid git repo. Always additive.
Schema Self-describing JSON Schema in the repo. Any future tool can parse every record without knowing g4a.
